前幾篇文章介紹了 Codex 的基本概念、環境準備、專案建立與介面控制。本篇開始進入研究工作坊的核心：如何把一個模糊的研究需求，寫成 Codex 能夠執行、檢查，而且最後可以由研究者複核的工作指令。

這篇文章是參考了codex官網的提示詞最佳實踐，並且針對資訊教育學門中可能需要處理以下工作的教授與研究者：

- 搜集數位學習、資訊素養、程式教育、教育科技或人機互動相關論文。
- 整理論文的作者、年份、研究方法、研究對象與主要發現。
- 讀取訪談逐字稿，依照研究者建立的 codebook 加上不同 tag。
- 保留原始引文、受訪者編號、標籤與判斷理由。
- 將一次性的 AI 對話，逐步整理成可以重複執行與檢查的研究流程。

核心觀念可以先濃縮成一句話：

> 好的研究 Prompt 不是把每個操作步驟寫得越長越好，而是清楚定義研究成果、可用證據、限制條件與完成標準，再讓 Codex 選擇有效率的執行方式。

## 一、把 Prompt 當成研究協議

一般人使用 AI 時，可能只會輸入：

```text
請幫我找資訊教育相關論文。
```

這個要求太寬，Codex 不知道「資訊教育」指的是什麼範圍、要找哪一段時間、哪些研究方法符合需求、哪些資料需要保留，也不知道什麼時候算完成。

研究者真正需要的不是一段看起來很完整的回答，而是一個可以被檢查的研究中間產物，例如：

- 一份有 DOI 或原始連結的論文候選清單。
- 一張去除重複、標註納入理由的文獻篩選表。
- 一份每一段逐字稿都保留來源位置的編碼資料。
- 一份將不確定案例獨立列出的人工複核清單。

因此，研究 Prompt 應該比較像一份小型研究協議，至少回答六個問題：

| 問題 | Prompt 中應該說明的內容 |
| --- | --- |
| 要完成什麼？ | 研究目標與要產出的成果 |
| 為什麼做？ | 研究問題、理論背景或使用情境 |
| 可以使用什麼？ | 資料夾、檔案、資料庫、網頁或外部工具 |
| 有哪些限制？ | 時間範圍、語言、納入條件、隱私與不可修改的檔案 |
| 什麼算完成？ | 欄位完整、證據可追溯、輸出檔案已產生 |
| 什麼時候要停下來？ | 缺資料、證據不足、需要人工判斷或風險過高 |

GPT-5.6 的官方建議也將目標、脈絡、限制與完成條件視為一個好的起點。這四個欄位足以讓多數研究任務先變得清楚，再依需要加入證據與審查規則。

## 二、六個欄位寫出研究型 Prompt

### 1. Goal：明確寫出研究成果

目標不要只寫「幫我分析」，而要說明分析要產生什麼。

比較模糊的寫法：

```text
請分析這些訪談資料。
```

比較好的寫法：

```text
請依照 codebook.md 對訪談逐字稿進行多重標籤編碼，輸出每個片段的
受訪者編號、原始引文、標籤、判斷理由與信心程度，讓研究者可以逐筆複核。
```

目標中要包含「動作」與「成果」：

- 動作：搜尋、整理、去重、篩選、編碼、比較、統計。
- 成果：CSV、Markdown 報告、文獻矩陣、編碼表、人工複核清單。

### 2. Context：告訴 Codex 研究脈絡

同樣是「資訊教育」，研究問題可能完全不同。例如：

- 國中生使用生成式 AI 學習程式設計的經驗。
- 教師如何培養學生的資訊素養。
- 大學生使用線上協作工具進行問題解決。
- AI 輔助學習環境中的認知負荷與學習策略。

Prompt 中可以補充：

- 研究領域與關鍵詞。
- 研究對象與教育階段。
- 研究方法，例如質性訪談、問卷、設計研究或混合方法。
- 研究問題或初步假設。
- 專案中哪些檔案是規則、哪些檔案是原始資料、哪些檔案是輸出位置。

例如：

```text
本研究關注高中生使用生成式 AI 輔助程式設計學習時的策略、困難與反思。
研究對象為完成兩週課程的高中生，資料來源是半結構式訪談。
```

脈絡不是背景作文，而是用來幫助 Codex 做範圍判斷。與目前研究問題無關的資料，不應該因為關鍵字相似就被納入。

### 3. Evidence：指定哪些內容可以當作證據

研究用途最需要避免的問題，是 Codex 把推測寫成事實。因此要在 Prompt 中明確規定證據來源。

論文蒐集可以要求：

- 只列出實際找到的論文。
- 每筆都保留 DOI、期刊頁面或資料庫連結。
- 找不到 DOI 時保留原始連結，不能自行補寫。
- 摘要或研究結果只在來源中有明確支持時才填寫。
- 無法確認的欄位使用「未找到」或「需人工確認」，不要猜測。

質性分析可以要求：

- 引文必須逐字取自原始逐字稿。
- 每個片段要保留檔名、受訪者編號與段落或行號。
- 標籤判斷要引用 codebook 中的定義。
- 如果有多個可能標籤，允許多重標籤並說明原因。
- 如果內容不足以判斷，列入人工複核，不要強行分類。

「請不要捏造」通常不夠具體。更有效的寫法是告訴 Codex：遇到證據不足時要填什麼、放到哪個檔案，以及是否要停止後續動作。

### 4. Constraints：寫出研究與資料限制

限制條件可以分成研究範圍、資料安全與操作權限三類。

#### 研究範圍

- 發表年份，例如 2018–2026。
- 語言，例如中文與英文。
- 研究對象，例如中學教師或大學生。
- 研究場域，例如正式課堂，不包含企業訓練。
- 排除條件，例如純技術效能論文或沒有教育情境的研究。

#### 資料安全

- 不修改 `raw/` 原始資料。
- 不將可識別個資寫入輸出檔。
- 使用受訪者代碼，不使用姓名、電子郵件或學號。
- 未經確認，不把逐字稿傳送到外部服務。
- 重要資料只在研究者允許的專案範圍內讀取。

#### 操作權限

- 先讀取資料並提出計畫，再開始大量修改或產生檔案。
- 允許在 `outputs/` 建立新檔案。
- 如果需要網路搜尋、MCP 或其他外部連線，先說明用途。
- 不執行會刪除原始資料的指令。

限制不是要讓 Codex 變得不能工作，而是把研究者原本放在腦中的規則寫出來，讓整個工作流程更容易重現與交接。

### 5. Output：先定義可檢查的輸出格式

如果只要求「整理成一份報告」，Codex 可能產出一篇流暢但難以核對的文字。研究工作通常更適合先產出結構化表格，再由表格生成摘要。

論文整理可以要求以下欄位：

```text
record_id, title, authors, year, venue, doi, source_url,
research_context, participants, method, key_findings,
inclusion_status, inclusion_reason, evidence_note, review_status
```

訪談編碼可以要求：

```text
transcript_id, participant_id, segment_id, source_location,
verbatim_quote, code_id, code_name, rationale, confidence,
review_status, reviewer_note
```

每個欄位最好在 Prompt 中說明用途。例如 `source_location` 要填檔名與段落編號，`confidence` 只能使用 `high`、`medium`、`low`，`review_status` 初始統一填 `needs_review`。

格式越明確，研究者越能在試算表中排序、篩選、抽樣複核，也越容易讓下一次的 Codex 讀取前一次成果。

### 6. Done when：定義完成條件與停止條件

「找到很多論文」不是完成條件，「所有資料都標籤完」也不一定是完成條件。可以改成更可驗證的規則：

- 每筆論文都有至少一個可回溯來源。
- DOI 重複的紀錄已合併，疑似重複則獨立列出。
- 每個納入或排除決定都有理由。
- 每個編碼片段都有來源位置與原始引文。
- 不確定的案例沒有被默默歸類，而是出現在人工複核清單。
- 原始資料未被修改，輸出檔案位於指定資料夾。

停止條件則可以寫成：

- 搜尋結果只有摘要，無法支持完整研究發現時，停止推論並標記缺少全文。
- 找不到符合條件的文獻時，回報搜尋範圍與可能原因，不要自行擴大範圍。
- codebook 沒有涵蓋某種新現象時，先建立候選標籤供研究者決定，不要直接改寫正式 codebook。
- 發現逐字稿含有可識別個資時，先停止輸出並通知研究者處理匿名化。
- 需要存取專案外資料或外部服務時，先提出核准要求。

有了停止條件，Codex 才知道什麼時候應該繼續、什麼時候要回報問題，而不是為了產出一個完整答案而自行猜測。

## 三、研究型 Prompt 的基本結構

下面這個結構可以當作工作坊講義中的通用模板：

```text
角色：你是資訊教育研究助理，負責協助整理研究資料，不取代研究者做最後判斷。

目標：
[說明這次要產出的研究成果]

研究脈絡：
[研究問題、研究對象、教育場域、時間範圍]

可用資料：
[資料夾、檔案、資料庫、網頁或外部工具]
[說明每份資料的角色：規則、原始資料、範例、輸出]

限制：
- [納入與排除條件]
- [格式、語言與欄位要求]
- [隱私、匿名化與原始檔案保護規則]
- [遇到需要外部連線或高風險操作時的核准規則]

證據要求：
- [哪些欄位必須有來源]
- [不能確認時要填什麼]
- [哪些內容不能自行推論]

輸出：
[檔案名稱、格式、欄位、摘要與人工複核清單]

完成條件：
- [可驗證的完成標準]

停止條件：
- [什麼情況要停下來、回報缺口或請研究者決定]
```

這個模板不是每次都要照抄。小任務可以縮短，長期研究專案則可以把不常變動的規則放進 `AGENTS.md`、`research-protocol.md` 或專案內的 skill，讓每段對話只保留本次任務真正不同的部分。

## 四、論文蒐集 Prompt 範例

以下是一個適合資訊教育研究的起始版本：

```text
角色：你是資訊教育研究助理，協助建立可供研究者篩選的文獻候選清單。

目標：
搜尋並整理 2019–2026 年間，探討高中生使用生成式 AI 輔助程式設計學習的
中文或英文研究。先建立候選清單，不要直接撰寫文獻回顧結論。

研究脈絡：
本研究關注學習策略、遇到的困難、教師支持與學生反思。優先納入有明確
教育場域、研究對象與研究方法的同行評審研究。

可用資料與工具：
- 使用目前可用的搜尋工具或我提供的資料庫匯出檔。
- 如果只能取得摘要或書目資料，請清楚標記資料層級。
- 不要把無法開啟或無法驗證的搜尋摘要當成完整研究證據。

限制：
- 納入高中教育或相近中等教育場域。
- 排除只有模型效能、沒有學習情境的技術論文。
- 保留中文與英文題名；作者、年份、期刊或會議名稱盡量完整。
- 不修改原始匯出檔，輸出到 outputs/literature-candidates.csv。

證據要求：
- 每筆紀錄都要有 DOI、期刊頁面或資料庫原始連結至少一項。
- 不要自行補寫 DOI、作者、研究結果或出版年份。
- 無法確認的欄位填「需人工確認」，並在 evidence_note 說明原因。
- DOI、標題與作者組合疑似重複時先合併，無法判斷的紀錄放到 review/possible-duplicates.csv。

輸出欄位：
record_id, title, authors, year, venue, doi, source_url, abstract_available,
education_context, participants, method, topic_tags, inclusion_status,
inclusion_reason, evidence_note, review_status

完成條件：
- 產生候選清單與疑似重複清單。
- 每筆紀錄有納入、排除或待確認狀態。
- 先回報搜尋範圍、查詢詞、來源與結果數量，再寫入檔案。

停止條件：
- 若來源不足以確認核心欄位，保留紀錄並標記，不要猜測。
- 若搜尋結果沒有符合條件的研究，回報檢查過的範圍與下一個可行選項。
- 在建立檔案前先提出計畫，等我確認後再執行大量整理。
```

這個 Prompt 沒有要求 Codex 一定要使用某一個固定網站或某一組工具，而是定義搜尋結果必須具備的證據與欄位。這樣做的好處是，未來更換資料庫、搜尋介面或 MCP 連接器時，研究品質標準仍然一致。

## 五、質性編碼 Prompt 範例

以下範例適合在研究者已經建立初版 codebook 後使用：

```text
角色：你是協助資訊教育質性研究的資料分析助理。你可以提出編碼判斷，
但不取代研究者的理論詮釋與最後決定。

目標：
讀取 transcripts/anonymized/ 中的訪談逐字稿，依照 codebook.md 對每個有
研究意義的片段進行編碼。允許同一片段同時有多個標籤。

研究脈絡：
本研究探討教師使用生成式 AI 設計課程時的實作經驗、困難、支持需求與
專業判斷。編碼單位是一個可以表達完整意思的語句或連續段落，不要把
一句話切到失去上下文。

資料角色：
- codebook.md：正式的標籤定義、納入條件、排除條件與範例。
- transcripts/anonymized/：已匿名化的原始逐字稿，只讀取，不修改。
- outputs/：編碼結果與分析備忘錄。
- review/：需要研究者決定的模糊案例與候選新標籤。

限制：
- 保留 participant_id、transcript_id、segment_id 與來源位置。
- verbatim_quote 必須逐字取自逐字稿，不要把摘要當成引文。
- 只能使用 codebook.md 已定義的標籤；遇到新現象先列為 candidate_code。
- 同一段可以有多個正式標籤，但每個標籤都要有獨立判斷理由。
- 不修改原始逐字稿，不輸出姓名、學號、電子郵件或其他可識別資訊。

輸出欄位：
transcript_id, participant_id, segment_id, source_location, verbatim_quote,
code_id, code_name, rationale, confidence, review_status, reviewer_note

判斷規則：
- confidence 只能填 high、medium 或 low。
- 無法依 codebook 判斷時，保留原始片段並填 review_status=needs_review。
- 如果發現候選新標籤，另外輸出 review/candidate-codes.csv，包含片段、候選名稱、
  建議定義與形成原因，但不要直接修改 codebook.md。

完成條件：
- 每個輸出片段都有來源位置與原始引文。
- 每個正式標籤都能對應到 codebook 的定義。
- 所有 low confidence、candidate_code 與可能重疊的案例都被列入複核清單。
- 回報檔案數量、編碼片段數量、各標籤數量與待複核數量。

停止條件：
- 若發現資料尚未匿名化，停止批次輸出並通知我。
- 若 codebook 缺少必要定義，先列出需要研究者決定的問題。
- 先使用少量片段試跑並回報格式，確認後再處理全部資料。
```

這個範例特別保留了三個研究判斷空間：多重標籤、低信心案例，以及候選新標籤。AI 可以協助大量整理，但研究者仍然掌握 codebook 的正式變更權與最終詮釋權。

## 六、何時先讓 Codex 做計畫

論文蒐集與訪談編碼都不是適合一開始就執行到底的任務。當任務具有以下特徵時，先要求 Codex 只做計畫：

- 研究問題仍然模糊。
- 資料來源很多，且彼此格式不同。
- 需要外部搜尋、MCP 或其他連接器。
- 會產生大量輸出檔案。
- 涉及個資、研究倫理或敏感逐字稿。
- codebook 還可能變動。

可以先輸入：

```text
先不要修改任何檔案，也不要處理全部資料。
請先檢查專案結構，提出：
1. 你理解的研究目標；
2. 需要使用的資料與工具；
3. 可能的資料品質或隱私問題；
4. 預計產出的檔案與欄位；
5. 需要我先決定的問題。
等我確認計畫後，再處理一小批資料試跑。
```

這不是多一道形式上的步驟，而是把「研究設計」和「資料處理」分開。若第一輪計畫就發現納入條件不清楚、逐字稿沒有匿名化或輸出格式不適合，修正計畫的成本遠低於重新整理整批資料。

## 七、如何避免 Prompt 越寫越長

GPT-5.6 的官方建議之一，是先從已經能工作的 Prompt 開始，逐組移除重複指令與不相關工具，再用相同案例檢查結果是否變差。這個原則很適合研究工作坊，因為研究者往往會把所有提醒都貼進每一次對話，最後反而難以知道哪一條規則真正有效。

可以把規則分成三層：

| 層級 | 放置內容 | 例子 |
| --- | --- | --- |
| 專案規則 | 每次工作都不變的原則 | 原始資料不可修改、輸出放在 `outputs/` |
| 工作流程規則 | 某類任務反覆使用的流程 | 論文篩選欄位、質性編碼欄位 |
| 本次 Prompt | 這一次才不同的條件 | 研究主題、年份、特定 codebook |

專案規則可以放在 `AGENTS.md` 或 `research-guidelines.md`；重複使用的工作流程則可以逐步整理成 Codex skill。當教授每次都在複製同一段「請保留原始引文、標記低信心、不要修改原檔」時，就表示這些規則已經值得被整理成可重複使用的資源。

不過，skill 也不應該一開始就包辦所有研究情境。先用兩三個真實案例驗證一個小流程，確認輸入、輸出與複核方式穩定後，再把它封裝起來，通常比一開始建立一個涵蓋所有研究方法的大型 skill 更可靠。

## 八、把「完成」和「正確」分開

Codex 說「已完成」只代表它完成了被要求的操作，不代表研究結果已經通過學術判斷。研究工作至少要分成兩個階段：

1. **操作完成**：檔案已建立、欄位已填寫、資料已去重或標籤已產生。
2. **研究確認**：來源真的支持內容、標籤符合 codebook、模糊案例已人工複核。

Prompt 中可以要求 Codex 提供檢查摘要：

```text
完成處理後，請不要只回覆「完成」。
請回報：處理了哪些檔案、產生哪些輸出、哪些欄位缺失、哪些案例需要人工複核，
以及你沒有自行判斷的部分。
```

這種回報格式能讓研究者快速找到需要花時間的地方，也能避免「輸出檔案存在」被誤認為「分析品質已經足夠」。
