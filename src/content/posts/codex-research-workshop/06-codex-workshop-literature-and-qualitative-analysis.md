如果要為資訊教育學門的教授準備 Codex 工作坊，最有效的方式通常不是從程式語法開始，而是從他們已經熟悉的研究工作開始：找文獻、整理研究脈絡、閱讀逐字稿、建立編碼規則，以及把大量中間資料整理成可以寫進論文的證據。

本篇把工作坊設計成一個可實際操作的研究流程，聚焦兩個案例：

1. 搜集與整理資訊教育相關論文。
2. 依照 codebook 對質性訪談資料加上不同 tag。

兩個案例看似不同，但共用同一套方法：先建立清楚的研究專案，再把原始資料、規則、產出與人工複核分開；先讓 Codex 在小範圍試跑，確認格式與判斷原則，再擴大到整批資料。

> Codex 可以協助研究者搜尋、整理、編碼與產出中間資料，但不應該取代研究者決定研究問題、修改正式 codebook、判斷證據是否足夠，或做最後的理論詮釋。

## 一、工作坊結束時，教授應該帶走什麼

一場面向研究者的 Codex 工作坊，不應只以「大家成功問出一個答案」作為成果。更適合的完成標準，是每位參與者都建立一個小型研究專案，並產生以下成果：

- 一份研究問題與搜尋範圍說明。
- 一份可重複使用的論文搜尋 Prompt。
- 一份有來源、去重與納入理由的文獻候選表。
- 一份匿名化的逐字稿樣本。
- 一份包含定義、納入條件、排除條件與範例的 codebook。
- 一份保留原始引文與來源位置的多重標籤編碼表。
- 一份需要研究者人工複核的清單。
- 一個對研究者來說足夠清楚的下一步工作計畫。

這些成果比一篇看起來完整的 AI 生成摘要更重要，因為它們能被研究者打開、檢查、修正，也能在下一次工作中繼續使用。

## 二、建立研究專案資料夾

在 Codex 中，一個研究專案最好不要只是一個放滿檔案的資料夾，而是讓不同資料角色有清楚位置。可以先建立以下結構：

```text
info-ed-research/
├─ AGENTS.md
├─ README.md
├─ research-question.md
├─ search-protocol.md
├─ codebook.md
├─ sources/
│  ├─ raw/
│  ├─ cleaned/
│  └─ papers.csv
├─ transcripts/
│  ├─ raw/
│  └─ anonymized/
├─ analysis/
│  ├─ memos/
│  └─ decisions.md
├─ outputs/
│  ├─ literature-candidates.csv
│  └─ coded-segments.csv
└─ review/
   ├─ possible-duplicates.csv
   ├─ needs-review.csv
   └─ candidate-codes.csv
```

這個結構的重點不是資料夾名稱本身，而是區分四種角色：

| 資料角色 | 內容 | Codex 的預設行為 |
| --- | --- | --- |
| 規則 | 研究問題、搜尋協議、codebook | 讀取並遵守，不任意改寫 |
| 原始資料 | 原始論文匯出、原始逐字稿 | 只讀取，不直接修改 |
| 中間產物 | 清理表、編碼表、分析備忘錄 | 可以建立與反覆修正 |
| 人工決策 | 複核清單、研究決策紀錄 | 提供給研究者判斷與確認 |

### `AGENTS.md` 在研究專案中的用途

雖然 `AGENTS.md` 常被工程師用來放開發規範，但研究專案也可以用它保存不常變動的工作原則，例如：

```markdown
# 研究專案工作規則

- 使用繁體中文撰寫說明與分析備忘錄。
- `sources/raw/` 與 `transcripts/raw/` 只讀取，不修改或刪除。
- 所有輸出檔案放在 `outputs/`，待人工判斷的案例放在 `review/`。
- 不要捏造 DOI、作者、研究結果、引文或受訪者資訊。
- 逐字稿只使用 participant_id，不輸出姓名、學號、電子郵件等個資。
- 不確定的判斷標記為 `needs_review`，不要自行猜測。
- 完成工作後回報輸入檔案、輸出檔案、缺失欄位與待複核項目。
```

將這些規則放在專案層級，可以減少教授每一次都要重新貼上同一段提醒，也能讓不同研究助理使用同一個專案時遵循相同的基本原則。

## 三、工作坊第一階段：先處理研究倫理與資料安全

在讀取研究資料以前，應該先花時間說明資料安全。尤其是訪談逐字稿可能包含姓名、學校、職稱、地點、特殊事件或其他可以回推出身分的資訊。

### 先確認三件事

1. **研究倫理**：研究計畫與受訪者同意書是否允許使用 AI 或外部工具處理資料？
2. **資料治理**：哪些資料可以放入目前的 Codex 專案？哪些資料必須留在校內或特定環境？
3. **匿名化狀態**：逐字稿是否已移除直接識別資訊與容易重新識別的組合資訊？

如果答案不清楚，不要直接把完整逐字稿貼進對話。可以先：

- 使用研究者指定的匿名化流程處理資料。
- 將姓名、電子郵件、電話、學號與學校名稱替換成代碼。
- 將原始資料放在唯讀或權限受限的位置。
- 先用人工製作的短篇示例測試編碼格式。
- 與校內研究倫理或資訊安全單位確認可使用的工具與方案。

匿名化不只是用搜尋取代姓名。若受訪者談到一個只有一個人經歷過的事件，或同時透露學校、職位與時間，也可能仍然可以被辨認。Codex 可以協助找出疑似識別資訊，但最後的匿名化判斷仍然要由研究者負責。

## 四、案例一：搜集資訊教育相關論文

### 1. 先寫搜尋協議，再開始搜尋

教授常常已經有研究方向，但還沒有把搜尋範圍寫成其他人也能理解的規則。工作坊可以先建立 `search-protocol.md`：

```markdown
# 文獻搜尋協議

## 研究問題
高中生使用生成式 AI 輔助程式設計學習時，會採取哪些學習策略，
又會遇到哪些困難與教師支持需求？

## 納入條件
- 2019–2026 年發表。
- 中文或英文。
- 研究對象為高中生或相近的中等教育學生。
- 有明確的教育場域與研究方法。
- 主題涉及生成式 AI、程式設計學習或學習支持。

## 排除條件
- 只有模型效能、沒有教育情境的技術論文。
- 沒有可確認來源的搜尋摘要。
- 與中等教育學習者無關的企業訓練研究。

## 必備欄位
title, authors, year, venue, doi, source_url, abstract,
education_context, participants, method, key_findings,
inclusion_status, inclusion_reason, evidence_note
```

這份協議的用途，是讓「相關」變成可討論、可修正的條件，而不是讓 Codex 依照關鍵字自行猜測。

### 2. 決定資料從哪裡來

論文蒐集的品質取決於可用的搜尋工具與資料來源。Codex 可以使用目前環境中提供的瀏覽、搜尋、檔案或 MCP 連接器；如果沒有合適的外部連線，也可以先由教授從學校資料庫或其他允許的來源匯出 CSV，再交給 Codex 清理與篩選。

可以把來源分成三個層級：

| 層級 | 內容 | 可以支持什麼 |
| --- | --- | --- |
| 書目資料 | 題名、作者、年份、期刊、DOI | 確認論文是否存在與基本資料 |
| 摘要 | 摘要文字與來源連結 | 初步判斷主題、方法與研究對象 |
| 全文 | PDF、HTML 或全文資料 | 詳細研究方法、樣本、結果與限制 |

不要把只有標題的紀錄當成已經讀過全文，也不要把搜尋結果頁面的摘要當成完整研究結論。資料層級應該直接寫在輸出欄位或 `evidence_note` 中。

如果論文資料來自專案外部，MCP 或其他連接器可能讓 Codex 直接使用工具，而不需要研究者反覆複製貼上資料。實際上只需要加入能解除明確工作瓶頸的連接器，不必一開始就連接所有服務。

### 3. 先搜尋，再建立書目表

工作坊可以用以下 Prompt 開始：

```text
請先閱讀 search-protocol.md 與 README.md，不要修改 sources/raw/。

第一階段只做搜尋規劃：
1. 說明你理解的研究問題與納入條件。
2. 提出 3–5 組短而具辨識度的搜尋詞組，包含同義詞與中英文變體。
3. 說明每組搜尋詞要捕捉哪一類研究。
4. 列出你需要的來源與可能限制。
5. 提出完成後的輸出欄位與人工複核點。

先不要處理全部結果，等我確認搜尋規劃後，再試跑一小批資料。
```

這個階段不急著產出論文清單，而是先確認 Codex 是否理解研究問題。若它把「生成式 AI 輔助程式設計學習」誤解成「所有 AI 程式」，就應該先修正研究脈絡，而不是等結果出來才大量刪除。

### 4. 建立候選文獻表

確認搜尋策略後，再讓 Codex 處理小批結果：

```text
請依照已確認的 search-protocol.md，處理目前提供的第一批書目資料。

請完成：
- 統一欄位名稱與年份格式。
- 清理空白與明顯的編碼問題。
- 以 DOI 優先、標題與作者組合為輔進行去重。
- 為每筆紀錄填入 inclusion_status 與 inclusion_reason。
- 無法確認的欄位填「需人工確認」，不要補猜。
- 把疑似重複但無法確定的紀錄寫入 review/possible-duplicates.csv。

輸出到 outputs/literature-candidates.csv，並回報：
處理筆數、去除重複筆數、納入筆數、排除筆數、待確認筆數與主要缺口。
```

第一批結果要由研究者檢查，特別是以下內容：

- Codex 是否把非教育情境納入？
- `inclusion_reason` 是否真的對應搜尋協議？
- DOI 是否被錯誤合併或拆開？
- 摘要內容是否被誤寫成研究結果？
- 待確認案例是否真的被保留？

這是一個很適合在工作坊現場示範的複核點。教授不只看到 Codex 產出表格，也會看到如何發現規則不夠精確，再回頭修正協議。

### 5. 從候選清單進入文獻篩選

書目表完成後，可以把工作拆成兩輪：

1. **初篩**：根據題名、摘要與基本條件標記納入、排除或待確認。
2. **全文整理**：只針對初篩保留的論文整理研究方法、樣本、情境、主要發現與限制。

這樣拆分可以避免 Codex 在還沒有確認相關性的情況下，對所有論文進行昂貴又容易出錯的全文分析。

全文整理的輸出可以加入：

```text
full_text_available, research_design, sample_size, learning_context,
data_collection, analysis_method, key_findings, limitations,
relevance_to_question, evidence_location, reviewer_status
```

`evidence_location` 很重要。它可以記錄頁碼、章節、段落或原始連結，讓研究者在寫作時回到來源核對，而不是只依賴 Codex 的摘要。

## 五、案例二：對質性訪談加上多重標籤

### 1. 先理解 codebook 的角色

Codebook 不是一張只有標籤名稱的清單。至少要包含：

| 欄位 | 說明 |
| --- | --- |
| `code_id` | 穩定且不易重複的代碼，例如 `SUPPORT_01` |
| `code_name` | 研究者看得懂的標籤名稱 |
| `definition` | 這個標籤要捕捉的概念 |
| `include_when` | 什麼情況應該使用 |
| `exclude_when` | 什麼情況不應使用 |
| `positive_example` | 一段符合定義的例子 |
| `negative_example` | 一段看似相近但不符合的例子 |
| `related_codes` | 容易混淆或可能共現的標籤 |
| `version` | 這份 codebook 的版本與日期 |

例如：

```markdown
## SUPPORT_01 教師同儕支持

definition: 受訪者描述同事、社群或共同備課夥伴提供的具體協助。
include_when: 包含共同備課、分享教材、示範操作或提供情緒支持。
exclude_when: 只有學校行政命令，沒有同儕互動或協助。
positive_example: 「我會先問同領域的老師，他們會把上次用過的教材給我。」
negative_example: 「學校規定每位老師都要使用這個平台。」
related_codes: SUPPORT_02 行政支持、BARRIER_01 時間不足
```

定義越清楚，Codex 越能在相近概念間做出一致的初步判斷；研究者也越容易檢查某個標籤是否被過度使用。

### 2. 先用少量資料試跑

不要一開始就讓 Codex 讀取全部訪談。建議選擇一到兩份代表性逐字稿，或每位受訪者抽取一小段，先測試：

- 編碼單位是否太短或太長。
- 同一片段是否需要多個標籤。
- `verbatim_quote` 是否保留原文。
- 來源位置是否足以讓研究者找回原段落。
- `rationale` 是否引用 codebook，而不是只寫主觀感受。
- 低信心案例是否被標記為待複核。

試跑時可以使用：

```text
請先只處理 transcripts/anonymized/pilot-01.txt 與 pilot-02.txt。
不要修改 codebook.md，也不要處理其他逐字稿。

先回報你如何判斷編碼單位、如何處理多重標籤、哪些片段會列入 needs_review。
接著產出一份試跑結果，讓我先檢查欄位與判斷方式。
```

如果試跑結果顯示 `BARRIER_01 時間不足` 幾乎出現在每段話中，就代表定義可能太寬，或研究問題與編碼單位需要重新討論。先修正規則，再擴大批次，會比最後才處理整份過度編碼的資料更省時間。

### 3. 建立多重標籤編碼表

確認試跑格式後，輸出可以採用長格式，也就是同一個片段若有三個標籤，就產生三列資料：

```text
transcript_id, participant_id, segment_id, source_location,
verbatim_quote, code_id, code_name, rationale, confidence,
review_status, reviewer_note
```

例如：

| segment_id | verbatim_quote | code_name | confidence | review_status |
| --- | --- | --- | --- | --- |
| P03-S014 | 我會先問同領域的老師，也會一起修改教材。 | 教師同儕支持 | high | needs_review |
| P03-S014 | 我會先問同領域的老師，也會一起修改教材。 | 共同備課 | medium | needs_review |

同一片段重複出現是刻意的，因為長格式比較適合後續統計每個 tag 的出現次數、比較不同受訪者，或進行主題矩陣分析。

### 4. 把「不確定」變成資料

質性分析最忌諱 Codex 為了讓每一段都有標籤而強行分類。研究者可以要求它把不確定性明確寫出來：

- `confidence=high`：內容明確符合 codebook 定義，且沒有明顯競爭標籤。
- `confidence=medium`：大致符合，但需要結合上下文或研究者判斷。
- `confidence=low`：可能符合，也可能是相近概念，建議人工複核。
- `review_status=needs_review`：需要研究者確認、修正或決定是否建立新標籤。

另外建立 `review/candidate-codes.csv`：

```text
candidate_id, transcript_id, segment_id, source_location,
verbatim_quote, proposed_name, why_existing_codes_are_insufficient,
possible_definition, researcher_decision
```

候選新標籤不應直接寫回正式 `codebook.md`。正式 codebook 的修改應該由研究者依照研究問題、理論脈絡與團隊討論決定，並更新版本與決策紀錄。

## 六、人工複核不是最後才做的形式

Codex 的輸出需要人工複核，但複核不必等到整批處理完才開始。可以安排三個檢查點：

### 檢查點一：研究設計

確認 Codex 是否理解：

- 研究問題與對象。
- 納入與排除條件。
- 編碼單位與 codebook 定義。
- 哪些檔案可以讀取、哪些檔案不能修改。

### 檢查點二：小批試跑

抽查：

- 5–10 筆論文。
- 2–3 位受訪者的部分片段。
- 所有 low confidence 或待複核資料。

研究者要檢查的是判斷規則是否一致，不只是文句是否流暢。

### 檢查點三：整批完成

完成後檢查：

- 每筆資料是否有來源位置。
- 原始資料是否保持不變。
- 標籤數量是否異常集中。
- 是否有大量空欄、重複或無法辨識的資料。
- 待複核清單是否真的包含不確定案例。
- 摘要或報告中的數字能否回到原始表格。

可以要求 Codex 在最後產生一份 `analysis/quality-report.md`：

```text
請建立 analysis/quality-report.md，包含：
- 讀取與處理的檔案清單。
- 產生的輸出檔案與資料筆數。
- 缺失值、重複值與格式問題。
- 各論文納入狀態或各質性標籤的出現次數。
- low confidence、needs_review 與 candidate_code 的數量。
- 你沒有自行判斷的部分。
- 建議研究者下一步人工檢查的項目。
```

## 七、從一次性工作變成可重複流程

當教授完成一次論文整理或訪談編碼後，不要只保留最後的 CSV。還應該保存：

- 使用過的 Prompt。
- 當時的搜尋協議或 codebook 版本。
- Codex 使用的模型、推理設定與日期。
- 輸入資料範圍與檔案清單。
- 研究者修改過哪些標籤或納入判斷。
- 哪些案例曾經讓 Codex 犯錯。
- 下一次需要改進的規則。

這些紀錄可以寫在 `analysis/decisions.md`：

```markdown
# 研究決策紀錄

## 2026-07-13：調整 SUPPORT_01

- 原因：Codex 將行政公告也標記為同儕支持。
- 調整：在 exclude_when 加入「沒有同儕互動或協助」。
- 影響：重新複核第一批與所有 low confidence 案例。
- 決定者：研究團隊討論後確認。
```

如果某個流程已經穩定重複使用，例如「讀取逐字稿並輸出多重標籤表」，就可以進一步整理成 Codex skill。skill 應該只負責一個清楚的工作，定義輸入、輸出與觸發方式，不要一開始就試圖包辦整個質性研究方法。

同樣地，若論文清單來自校內資料庫、研究團隊常常需要定期更新，也可以考慮用 MCP 或排程工作流連接外部資料。但應該先讓手動流程穩定，再自動化；自動化的是已經被研究者理解和驗證過的方法。

## 八、工作坊可以怎麼安排

一場半天工作坊可以安排成四個階段：

| 時間 | 主題 | 參與者產出 |
| --- | --- | --- |
| 30 分鐘 | Codex 專案、權限與資料安全 | 建立專案結構與匿名化檢查表 |
| 45 分鐘 | 研究型 Prompt 設計 | 完成研究問題、搜尋協議或 codebook 初稿 |
| 60 分鐘 | 論文蒐集案例 | 產出候選文獻表與待確認清單 |
| 60 分鐘 | 質性編碼案例 | 產出試跑編碼表與多重標籤複核清單 |
| 30 分鐘 | 驗證與流程封裝 | 完成 quality report 與下一步工作計畫 |

如果時間有限，建議不要同時處理太多主題。可以讓每位教授帶一個實際研究問題，使用少量、已匿名化或可公開的資料完成一個小閉環：

```text
研究問題 → 規則文件 → 小批試跑 → 人工複核 → 調整規則 → 產出報告
```

學員真正學會的不是某一個固定 Prompt，而是如何在每個階段判斷「現在的結果是否足以進入下一步」。

## 九、常見失敗與處理方式

### 只說「找相關論文」

問題：研究範圍、時間、對象與證據層級不清楚。

改進：先寫搜尋協議，指定納入、排除、必備欄位與來源要求。

### 直接把所有逐字稿交給 Codex

問題：可能有個資、codebook 尚未穩定，也沒有試跑與複核點。

改進：先匿名化、建立 codebook、用少量資料試跑，再擴大處理。

### 要 Codex 自己發明標籤

問題：標籤可能失去研究設計的一致性，研究者也難以解釋其形成過程。

改進：正式標籤依 codebook，新的現象先放入 candidate codes，交由研究者決定。

### 只看最後摘要，不看中間表格

問題：研究者無法追蹤來源、引文、判斷理由與不確定性。

改進：先產出結構化資料與人工複核清單，再由表格生成摘要。

### 為了方便直接開完整權限

問題：研究資料與檔案可能被不必要地讀取、修改或傳送。

改進：先使用限制較小的沙箱與核准模式，只有在明確知道需求時才放寬權限。

### 把 Codex 的「完成」當成研究結論

問題：操作完成不等於證據充分，更不等於學術詮釋成立。

改進：把檔案產出、證據核對、標籤複核與研究者決策分成不同階段。

## 最後的想法

Codex 對資訊教育教授最有價值的地方，不只是替研究者節省打字時間，而是可以協助把大量、重複、需要結構化的研究工作整理成一套可追蹤流程。

論文蒐集時，它可以協助建立候選清單、整理欄位、去除重複、標註證據與篩選理由；質性分析時，它可以依照 codebook 切分片段、套用多個 tag、保留原始引文、說明判斷理由，並把低信心案例交回研究者。

但可靠的研究工作流必須保留人的位置：研究者決定問題與範圍，研究團隊維護 codebook，人工複核證據與標籤，最後再由研究者進行理論詮釋與學術寫作。

當「研究問題、資料、規則、輸出與複核」都被清楚分開，Codex 才會從一次性的聊天工具，變成研究團隊可以共同使用、持續改善的研究助理。
