前幾篇文章已經示範如何使用 Codex 搜集論文、整理文獻，以及依照 codebook 對訪談逐字稿加上不同標籤。但如果每次開啟新對話，都要重新貼上「不要修改原始資料」、「輸出放到 outputs/」、「不確定時要交給研究者複核」，工作很快就會變得重複又容易遺漏。

`AGENTS.md` 就是用來保存這些長期工作規則的檔案。

你可以把它想成研究專案給 AI 助理的「工作手冊」：它不負責回答某一次的研究問題，而是告訴 Codex，在這個專案中應該如何讀取資料、如何修改檔案、如何驗證結果，以及哪些事情不能做。

## 一、AGENTS.md 是什麼

AGENTS.md 是一個普通的 Markdown 檔案。當 Codex 開始工作時，會在符合條件的目錄中尋找這類指引，並把內容帶入工作上下文，讓後續任務遵循專案規則。

它適合放置「長期不變，而且每次工作都可能需要」的資訊，例如：

- 專案資料夾的用途。
- 如何執行分析、測試或產生報告。
- 原始資料和輸出資料的分工。
- 研究團隊使用的命名與格式規則。
- 隱私、匿名化與資料安全限制。
- 什麼情況需要人工確認。
- 完成任務前必須進行的檢查。

例如，不要每次都在 Prompt 裡重複：

```text
請不要修改 raw/ 裡的檔案，輸出放到 outputs/，不確定的內容放到 review/。
```

如果這是整個研究專案的固定規則，就應該寫進 `AGENTS.md`。之後每段新對話只需要補充本次研究問題、資料範圍與輸出目標。

## 二、AGENTS.md 和其他 Codex 功能有什麼不同

Codex 有很多可以保存規則與工作方式的地方，初學者很容易混在一起。

| 工具或位置 | 適合保存什麼 | 研究專案例子 |
| --- | --- | --- |
| Prompt | 這一次任務的要求 | 搜尋 2020–2026 年的生成式 AI 教學研究 |
| `AGENTS.md` | 專案長期工作規則 | 原始資料不可修改、輸出格式、驗證方式 |
| `.codex/config.toml` | Codex 的設定與環境偏好 | 沙箱、模型、MCP、hooks 或權限預設 |
| Skill | 可重複執行的單一工作流程 | 將逐字稿依 codebook 編碼並輸出 CSV |
| MCP / Connector | 外部資料與工具連線 | 學校資料庫、雲端文件或研究資料服務 |

可以用一句話區分：

> `AGENTS.md` 說明「在這個專案中應該遵守什麼規則」；skill 說明「如何完成一種可重複工作」；MCP 提供「需要使用的外部資料或工具」；Prompt 則說明「這一次要完成什麼」。

不要把所有內容都塞進 `AGENTS.md`。一份短而準確的專案規則，通常比一份把所有研究方法、背景知識與每次任務都寫進去的超長檔案更有用。

## 三、Codex 如何找到 AGENTS.md

Codex 會建立一條 instruction chain，把不同層級的規則合併到目前工作中。理解搜尋順序，可以避免「我明明寫了規則，為什麼 Codex 沒有遵守」的問題。

### 1. Global scope：全域規則

Codex home 預設是 `~/.codex`，也可以透過 `CODEX_HOME` 指定其他位置。

在這個位置，Codex 會優先查看：

```text
~/.codex/AGENTS.override.md
~/.codex/AGENTS.md
```

如果 `AGENTS.override.md` 存在且有內容，會優先使用它；否則才使用 `AGENTS.md`。同一層級不會同時載入這兩份檔案。

全域指引適合放置個人長期偏好，例如：

- 偏好使用 `pnpm` 或某個 Python 工具。
- 回答與報告使用繁體中文。
- 修改程式後執行哪些基本檢查。
- 不要擅自新增生產環境套件。

但不建議把某一個研究計畫的資料夾、codebook 或受訪者規則放在全域檔案，因為這些內容不應該影響其他專案。

### 2. Project scope：專案規則

Codex 會從專案根目錄開始，沿著目前工作目錄往下尋找指引。專案根目錄通常是 Git repository 的根目錄。

例如：

```text
info-ed-research/
├─ AGENTS.md
└─ analysis/
   └─ coded-data/
```

如果 Codex 在 `info-ed-research/analysis/coded-data/` 中工作，它會先讀取專案根目錄的 `AGENTS.md`，再檢查中間路徑是否有更接近目前目錄的指引檔案。

### 3. Nested scope：子目錄規則

當不同資料夾有不同工作規則時，可以在子目錄放置更具體的 `AGENTS.md` 或 `AGENTS.override.md`。

例如：

```text
info-ed-research/
├─ AGENTS.md
├─ literature/
│  └─ AGENTS.md
└─ transcripts/
   └─ AGENTS.md
```

根目錄可以放整個研究專案共用的規則；`literature/AGENTS.md` 可以補充論文整理欄位；`transcripts/AGENTS.md` 則可以補充逐字稿匿名化與編碼限制。

越接近目前工作目錄的規則，越能針對特定子專案提供補充或覆寫。合併時，Codex 會按照從上到下的順序串接內容，因此較接近目前目錄的指引會出現在後面，能夠覆蓋較上層的模糊或衝突規則。

## 四、三種範圍如何配置

對教授與研究團隊來說，可以用以下方式分配：

| 範圍 | 放置位置 | 適合內容 |
| --- | --- | --- |
| 個人 | `~/.codex/AGENTS.md` | 個人語言、工具與確認習慣 |
| 研究專案 | `研究專案/AGENTS.md` | 研究資料結構、隱私、輸出與驗證 |
| 特定資料類型 | `研究專案/transcripts/AGENTS.md` | 逐字稿、codebook 與多重標籤規則 |
| 暫時覆寫 | `AGENTS.override.md` | 短期實驗或特殊任務的例外規則 |

一個安全的基本策略是：

1. 全域只放不涉及特定研究資料的個人偏好。
2. 研究倫理、原始資料與輸出規則放在研究專案根目錄。
3. 只有當子資料夾真的需要不同規則時，才建立 nested instructions。
4. 暫時例外使用 `AGENTS.override.md`，完成後移除或改回正式規則。

## 五、資訊教育研究專案的 AGENTS.md 範例

以下是一份可以放在資訊教育研究專案根目錄的起始範例。它不是所有研究都適用的固定答案，教授應該依照研究倫理審查、資料管理計畫與研究團隊習慣修改。

```markdown
# 資訊教育研究專案工作規則

## 專案目的

本專案用於整理資訊教育、數位學習與生成式 AI 教學研究，
包含文獻蒐集、訪談逐字稿分析與研究報告整理。

## 資料夾用途

- `research-question.md`：研究問題、研究對象與研究範圍。
- `search-protocol.md`：論文搜尋、納入與排除條件。
- `codebook.md`：正式質性編碼規則與版本紀錄。
- `sources/raw/`：原始書目匯出與原始來源，只讀取，不修改。
- `sources/cleaned/`：清理後的書目資料。
- `transcripts/raw/`：原始逐字稿，只讀取，不修改。
- `transcripts/anonymized/`：已匿名化、可供分析的逐字稿。
- `outputs/`：分析結果、表格與報告草稿。
- `review/`：需要研究者人工判斷的資料。
- `analysis/decisions.md`：研究團隊的分析決策紀錄。

## 研究資料安全

- 不修改、刪除或覆寫 `sources/raw/` 與 `transcripts/raw/`。
- 使用 `participant_id`，不要在輸出中使用姓名、學號、電子郵件或電話。
- 如果發現逐字稿可能尚未匿名化，先停止批次分析並通知研究者。
- 未經研究者確認，不將研究資料送往外部服務。
- 不要把 API key、密碼或其他秘密資訊寫入 Prompt、報告或版本控制。

## 論文整理規則

- 依照 `search-protocol.md` 判斷納入、排除或待確認。
- 不要捏造 DOI、作者、年份、研究方法或研究結果。
- 每筆論文保留 DOI、原始頁面或資料庫連結。
- 找不到或無法確認的欄位填入「需人工確認」。
- 疑似重複的資料放到 `review/possible-duplicates.csv`。

## 質性編碼規則

- 依照 `codebook.md` 使用正式標籤。
- 同一個片段可以有多個標籤，但每個標籤都要有判斷理由。
- `verbatim_quote` 必須逐字取自匿名化逐字稿。
- 保留 `transcript_id`、`participant_id`、`segment_id` 與來源位置。
- 無法依 codebook 判斷時，使用 `needs_review`，不要強行分類。
- 發現 codebook 沒有涵蓋的新現象時，放到 `review/candidate-codes.csv`，不要直接修改正式 codebook。

## 工作方式

- 複雜任務先提出計畫與需要研究者決定的問題。
- 先用少量資料試跑，經研究者確認格式後再處理全部資料。
- 不要為了讓輸出看起來完整而補猜缺失資訊。
- 完成後回報輸入檔案、輸出檔案、資料筆數、缺失欄位與待複核項目。

## 完成標準

- 原始資料保持不變。
- 所有輸出位於指定資料夾。
- 每筆資料都能回到來源位置。
- 不確定與需要人工判斷的案例已被獨立列出。
- 研究者可以檢查並說明 Codex 做過哪些整理與判斷。
```

這份檔案的重點不是讓 Codex 自動完成研究，而是讓每一次工作都遵守相同的最低品質標準。

## 六、為文獻與逐字稿建立子目錄規則

如果根目錄規則開始變得太長，可以將特定資料類型的規則下放。

### `literature/AGENTS.md`

```markdown
# 文獻資料規則

- 先閱讀根目錄的 `search-protocol.md`。
- 書目資料至少保留 `title`、`authors`、`year`、`venue` 與 `source_url`。
- DOI 不可由模型推測；無法確認時標記為 `needs_review`。
- 初篩與全文整理分成兩個階段。
- 先輸出結構化表格，再產生文獻摘要。
- 摘要中的研究發現必須可以回到論文或全文位置。
```

### `transcripts/AGENTS.md`

```markdown
# 訪談資料規則

- 只分析 `anonymized/`，不要讀取或修改 `raw/`。
- 先確認 `codebook.md` 的版本，再開始編碼。
- 編碼單位要保留足夠上下文，不要只截取單一關鍵詞。
- 允許多重標籤，使用長格式輸出。
- 每筆標籤都要保留原始引文、來源位置、判斷理由與信心程度。
- low confidence 與 candidate code 一律列入人工複核。
```

這種分層方式有兩個優點：根目錄保持簡短，特定資料夾的規則也不會污染其他工作。另一個好處是，未來研究團隊增加新的資料類型時，可以只新增對應的規則檔案。

## 七、AGENTS.md 應該寫什麼，不應該寫什麼

### 適合寫進去的內容

#### 專案結構

告訴 Codex 哪些檔案是研究問題、搜尋規則、codebook、原始資料與輸出資料。

#### 可重複的操作方式

例如：

- 使用哪個指令產生報告。
- 如何執行資料清理。
- 如何執行檢查或測試。
- 產出檔案應該放在哪裡。

#### 品質與安全規則

例如：

- 不修改原始資料。
- 缺失證據不能猜測。
- 個資必須匿名化。
- 低信心案例交給人工複核。

#### 完成與驗證條件

不要只要求「完成分析」，而要告訴 Codex 完成後要回報什麼，並且要如何檢查輸出。

### 不適合塞進去的內容

#### 每次任務才會改變的指令

例如「這次只分析 2024 年的論文」。這類條件應該留在本次 Prompt 或 `search-protocol.md` 的版本中。

#### 大量研究背景與全文教材

AGENTS.md 應該是工作規則，不是整本研究方法教科書。大型背景資料可以放在 `references/` 或專案文件，再在 AGENTS.md 指定 Codex 何時讀取。

#### 機密資訊

不要把密碼、API key、受訪者姓名、個資或內部憑證寫進 AGENTS.md。這個檔案可能會被提交到 Git repository，也可能會被每一段 Codex 對話讀取。

#### 模糊的口號

像是「請盡力完成」、「永遠做出最好的判斷」通常不如具體規則有效。改寫成「缺少 DOI 時標記 `needs_review`，不要自行補寫」會更容易檢查。

#### 與其他規則重複的長篇文字

同一條規則在全域檔案、專案檔案與每次 Prompt 重複出現，會增加上下文，也可能讓不同版本的規則彼此矛盾。把規則放在最適合的單一位置即可。

## 八、如何確認 Codex 真的讀到了規則

AGENTS.md 通常在 Codex 啟動或新工作階段建立 instruction chain 時讀取。修改檔案後，建議重新啟動新的 Codex 工作階段或建立新的對話，再進行確認。

在 CLI 中，可以使用：

```bash
codex --ask-for-approval never "請摘要目前載入的專案工作規則，並列出你不確定的部分。"
```

如果使用子目錄工作，也可以指定工作目錄：

```bash
codex --cd transcripts --ask-for-approval never "列出目前生效的逐字稿處理規則。"
```

在桌面 App 中，則可以直接開啟一段新任務，輸入：

```text
請先不要修改任何檔案。
請說明你目前理解的專案規則、原始資料限制、輸出位置與完成檢查方式。
如果有任何規則沒有被讀到或彼此衝突，請列出來。
```

不要只問「你有讀到 AGENTS.md 嗎？」。讓 Codex 摘要規則，比單純回答「有」更能驗證它是否真的理解了內容。

## 九、常見問題與排查方式

### Codex 完全沒有遵守 AGENTS.md

先檢查：

1. 檔案是否真的叫 `AGENTS.md`，而不是 `AGENTS.md.txt`。
2. 目前工作目錄是否位於預期的 Git repository。
3. `CODEX_HOME` 是否指向另一個 Codex 設定目錄。
4. 是否存在更接近目前目錄的 `AGENTS.override.md`。
5. 是否在修改檔案後仍然使用舊的工作階段。

### 只有某些資料夾的規則沒有作用

檢查目前的工作目錄與父層路徑，確認是否存在拼錯檔名、空白檔案或更高優先級的 override。也要確認規則是否被放在目前工作目錄的上層，而不是另一個平行資料夾。

### AGENTS.md 太長，後面的規則被忽略

Codex 會受 `project_doc_max_bytes` 限制。官方文件指出預設上限是 32 KiB；當規則太多時，可以：

- 刪除重複與不再使用的內容。
- 將特定資料類型的規則移到 nested `AGENTS.md`。
- 將大型參考資料放到其他文件，只有需要時才讀取。
- 在設定中調整 `project_doc_max_bytes`，但先確認增加上下文成本是否值得。

### 想使用其他檔名

如果研究團隊已經使用 `TEAM_GUIDE.md` 或 `.agents.md`，可以在 Codex 設定中使用 `project_doc_fallback_filenames` 加入其他檔名。這樣 Codex 才會把它們當成專案指引；單純把檔案命名成其他名稱，不會自動生效。

## 十、建立與維護 AGENTS.md 的建議流程

不要一開始就寫一份涵蓋所有情境的完美規格。比較實用的做法是：

1. **先寫最小版本**：專案結構、原始資料限制、輸出位置與完成標準。
2. **用一個真實任務試跑**：例如整理 20 篇候選論文或編碼兩份匿名逐字稿。
3. **記錄 Codex 的錯誤**：它是否讀錯資料夾、猜 DOI、過度使用標籤或忘記回報缺口？
4. **只新增能避免重複錯誤的規則**：不要把每一次偶然問題都變成永久規定。
5. **定期刪除過時規則**：研究問題、資料夾或工具改變時，同步更新 AGENTS.md。
6. **保留研究決策紀錄**：如果修改了 codebook 或資料處理方式，記錄原因與影響。

一份好的 AGENTS.md 不是越長越專業，而是能讓研究團隊少做重複說明、少犯相同錯誤，而且更容易檢查 Codex 的輸出。

## 十一、工作坊練習：建立自己的研究規則

在工作坊中，可以讓每位教授用自己的研究題目完成以下練習：

### 練習一：列出專案規則

請回答：

- 哪些資料是原始資料，絕對不能修改？
- 哪些檔案是研究規則或 codebook？
- 輸出應該放在哪裡？
- 哪些判斷必須交給研究者？
- 什麼結果才算完成？

### 練習二：把規則寫成 AGENTS.md

先不要寫超過一頁，完成以下五個區塊：

```markdown
# 專案工作規則

## 專案結構
## 資料安全
## 研究工作規則
## 輸出格式
## 完成與驗證
```

### 練習三：用 Codex 讀取與摘要

建立新任務，要求 Codex 先摘要它理解到的規則，並指出任何含糊或互相衝突的地方。接著讓它只處理一小批資料，觀察規則是否真的影響輸出。

### 練習四：依照實際錯誤改進

如果 Codex 把「行政命令」誤標成「同儕支持」，不要只在對話中糾正一次。檢查這是否是整個研究專案都會遇到的問題；如果是，就把更精確的排除條件補進 codebook 或 `AGENTS.md`，再重新試跑。

## 最後的想法

`AGENTS.md` 的價值，在於把研究團隊原本依賴經驗、口頭交代與每次 Prompt 重複說明的工作方式，整理成一份可以被 Codex 讀取的專案規範。

對資訊教育研究而言，它可以提醒 Codex：論文要保留來源與證據，逐字稿要先匿名化，正式標籤要依照 codebook，多重標籤要保留判斷理由，不確定的內容要交給研究者，原始資料不能被覆寫。

當研究規則被保存下來，Codex 才不會每次都像第一次加入研究團隊一樣重新學習。教授也可以把時間從「反覆提醒 AI」移回研究設計、證據判斷與理論詮釋本身。
